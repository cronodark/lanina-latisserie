# Plan: Perbaiki SweetAlert Tidak Muncul di Production

## Konteks & Akar Masalah

Error di production:
```
Uncaught ReferenceError: Swal is not defined
    at konfirmasiHapus (address:546:17)
```

**Setup sekarang:**
- `resources/js/app.js` mengimpor SweetAlert dan men-assign ke global:
  ```js
  import Swal from 'sweetalert2';
  window.Swal = Swal;
  ```
- Layout (`layouts/app.blade.php:9`, `layouts/admin.blade.php:8`) memanggil `@vite(['resources/css/app.css', 'resources/js/app.js'])`. Vite merender tag `<script type="module" src="/build/assets/app-xxx.js">`.
- Beberapa view memanggil `Swal.fire(...)` dari **inline `<script>` biasa**:
  - `components/swal.blade.php:94`
  - `pages/address/index.blade.php:128`
  - `pages/customer/belumBayar.blade.php`, `diproses.blade.php`, `diantar.blade.php`

**Dua penyebab yang saling menguatkan di production cPanel:**

1. **Race condition script module vs classic.** `<script type="module">` selalu implicit `defer` — baru dieksekusi setelah parsing HTML selesai. Inline `<script>` classic dieksekusi segera saat parser menemuinya. Di halaman dengan `x-swal` inline script di tengah body, urutannya: inline script jalan duluan → `window.Swal` masih `undefined` → `ReferenceError`. Di dev server biasanya lebih toleran karena HMR/modul ter-load lebih cepat, di production minified bundle besar jadi gap-nya lebih terasa.

2. **Build artefak kemungkinan hilang / path salah.** User memilih "Tidak yakin" untuk status build. Di shared hosting cPanel umum terjadi:
   - Folder `public/build/` tidak ter-upload lewat FTP (banyak orang lupa upload folder yang ter-gitignore).
   - `APP_URL` tidak match dengan URL asli sehingga `@vite` meng-emit path asset yang salah (mis. `http://localhost/build/...` saat host sebenarnya `https://domain.com`).
   - Symlink dari `public_html/` ke `public/` tidak ter-setup sehingga file di `public/build/` tidak ter-serve.

Kalau `window.Swal` tidak pernah ter-assign sama sekali (karena file JS-nya 404), semua pemanggilan `Swal.fire` pasti error. Kalau file-nya ada tapi ada race condition, error hanya muncul pada interaksi awal atau di halaman tertentu.

## Strategi Perbaikan (dua lapis)

Saya usulkan dua lapis perbaikan yang saling melengkapi, urutan dari yang paling cepat ke yang paling menyeluruh.

### Lapis 1: Fix race condition (code-level, tidak tergantung build)

Tambahkan **fallback CDN** di layout, sebelum inline script mana pun yang memanggil `Swal`:

```html
{{-- layouts/app.blade.php & layouts/admin.blade.php <head> --}}
@vite(['resources/css/app.css', 'resources/js/app.js'])
<script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>
```

Dengan ini:
- CDN memuat SweetAlert sebagai classic script di `<head>`, jadi sudah ter-eksekusi **sebelum** body diparse dan sebelum inline script mana pun jalan.
- `window.Swal` dijamin tersedia tanpa bergantung pada bundle Vite.
- Kalau bundle Vite juga sukses dimuat, `app.js` akan me-reassign `window.Swal = Swal` — objek dari Vite bundle menimpa objek CDN, tidak ada konflik karena API-nya identik.
- Kalau bundle Vite gagal (asset 404, hot-reload dev server tidak nyala), CDN tetap menyelamatkan.

Alpine.js di project ini sudah pakai pola yang sama (CDN di `<head>` defer), jadi ini konsisten.

**Alternatif (lebih "clean" tapi kurang resilient):** Bungkus semua pemanggilan `Swal.fire` di inline script dengan guard DOMContentLoaded + polling, mis.
```js
function waitForSwal(cb) {
    if (window.Swal) return cb();
    setTimeout(() => waitForSwal(cb), 50);
}
```
Pilih pendekatan ini kalau user anti-CDN. Tapi tambah kompleksitas dan tidak menyelesaikan kasus "bundle 404".

### Lapis 2: Verifikasi deploy & build (infra-level)

User perlu cek di cPanel:

1. **Pastikan `npm run build` sudah dijalankan lokal** sebelum upload, atau kalau hosting mendukung Node, jalankan di server. Hasil build harus menghasilkan:
   - `public/build/manifest.json`
   - `public/build/assets/app-*.js`
   - `public/build/assets/app-*.css`
2. **Upload folder `public/build/` ke server.** Banyak developer lupa karena `public/build` kadang di-gitignore. Cek `.gitignore` — kalau iya, harus upload manual via FTP/File Manager.
3. **Cek `APP_URL` di `.env` production** sama dengan URL asli (mis. `https://lanina.com`, bukan `http://localhost`). Setelah ubah: `php artisan config:clear`.
4. **Test langsung di browser**: buka `https://domain.com/build/manifest.json` dan `https://domain.com/build/assets/app-xxx.js`. Kalau 404, berarti folder build tidak ter-serve. Umumnya di cPanel, folder `public/` Laravel perlu dijadikan document root atau isinya di-symlink ke `public_html/`.

Kalau Lapis 1 sudah diterapkan, user masih bisa tetap fungsional walau Lapis 2 belum selesai, karena Swal akan di-load dari CDN.

## Rencana Implementasi (saat user approve)

1. **Edit `resources/views/layouts/app.blade.php`** — tambah `<script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>` setelah `@vite(...)`.
2. **Edit `resources/views/layouts/admin.blade.php`** — tambah CDN yang sama.
3. **Verifikasi** dengan membaca file setelah edit bahwa urutan load benar (CDN sebelum penggunaan Swal di inline script).
4. **Tulis dokumen singkat** (inline di chat) berisi checklist untuk user:
   - Jalankan `npm run build` lokal.
   - Upload `public/build/` ke server.
   - Cek `APP_URL` di `.env`.
   - Jalankan `php artisan config:cache` dan `php artisan view:clear` di server.
5. Tidak perlu edit file lain. Inline script di setiap view tetap jalan tanpa modifikasi karena `window.Swal` akan dijamin tersedia oleh CDN.

## File yang Akan Diubah

| File | Aksi |
|---|---|
| `resources/views/layouts/app.blade.php` | Tambah 1 baris CDN SweetAlert setelah `@vite` |
| `resources/views/layouts/admin.blade.php` | Sama |

## Risiko & Catatan

- **Dependency eksternal:** CDN jsdelivr sangat reliable (uptime > 99.99%), tapi kalau offline/blocked, Swal masih bisa jalan via Vite bundle. Redundansinya bagus.
- **Duplikasi download:** Browser akan download Swal 2x (CDN + bundle). Ukuran ~40KB gzipped. Trade-off: kehandalan vs bandwidth. Untuk aplikasi e-commerce dengan UX penting, worth it.
- **Bisa di-remove nanti** setelah user yakin Vite build stabil di production. Tinggal hapus 1 baris.

## Verifikasi Post-implementasi

User test di production:
1. Buka halaman address list.
2. Klik tombol hapus alamat.
3. Modal SweetAlert muncul.
4. Cek Console — tidak ada error `Swal is not defined`.
5. Test juga di halaman customer (belumBayar/diproses/diantar) untuk konfirmasi.

# (Historis) Plan: Validasi Ketat Form Alamat Customer

## Tujuan

Mencegah customer mengisi alamat dengan data asal-asalan dengan cara:
1. Menggunakan dropdown cascading untuk Provinsi → Kabupaten → Kecamatan via API wilayah Indonesia (emsifa).
2. Memastikan RT/RW hanya menerima input angka dengan format 3 digit sesuai schema DB (`char(3)`).
3. Memperketat validasi field lain (kode pos, alamat jalan) dengan regex pattern.

## Konteks Saat Ini

- **Form view**: [`resources/views/pages/address/create.blade.php`](resources/views/pages/address/create.blade.php) dan [`resources/views/pages/address/edit.blade.php`](resources/views/pages/address/edit.blade.php)
- **Controller**: [`app/Http/Controllers/AddressController.php`](app/Http/Controllers/AddressController.php) — validasi sekarang hanya `required|string|max:n` sehingga user bisa isi apa saja.
- **Schema DB** (`2026_04_23_022552_create_addresses_table.php`):
  - `zip_code` = `char(5)`
  - `rt` = `char(3)`
  - `rw` = `char(3)`
  - `state`, `city`, `district`, `street` = `string`
- **API sumber**: `https://emsifa.github.io/api-wilayah-indonesia/api/`
  - Provinces: `GET /provinces.json`
  - Regencies: `GET /regencies/{province_id}.json`
  - Districts: `GET /districts/{regency_id}.json`

## Rancangan Perubahan

### 1. AddressController — `addressRules()`

Perketat aturan validasi:

```php
return [
    'street'   => ['required', 'string', 'min:5', 'max:255'],
    'district' => ['required', 'string', 'max:255'],
    'city'     => ['required', 'string', 'max:255'],
    'state'    => ['required', 'string', 'max:255'],
    'zip_code' => ['required', 'digits:5'],
    'rt'       => ['required', 'digits_between:1,3'],
    'rw'       => ['required', 'digits_between:1,3'],
    'notes'    => ['nullable', 'string', 'max:500'],
];
```

Tambahkan pesan custom di `addressValidationMessages()`:

```php
'digits'         => 'Kolom :attribute harus berupa :digits digit angka.',
'digits_between' => 'Kolom :attribute harus berupa angka dengan panjang :min sampai :max digit.',
'min'            => 'Kolom :attribute minimal :min karakter.',
```

Tambahkan logic normalisasi di `store()` dan `update()` agar RT/RW disimpan sebagai `str_pad(..., 3, '0', STR_PAD_LEFT)` supaya selalu pas `char(3)` (mis. user isi `1` → tersimpan `001`).

### 2. View `create.blade.php` dan `edit.blade.php`

#### a. Ubah Provinsi, Kabupaten, Kecamatan jadi `<select>` cascading

Bungkus 3 field ini dalam Alpine component:

```html
<div x-data="addressWilayah({
        initStateId:    '{{ old('state_id', $address->state_id ?? '') }}',
        initCityId:     '{{ old('city_id', $address->city_id ?? '') }}',
        initDistrictId: '{{ old('district_id', $address->district_id ?? '') }}',
        initStateName:    '{{ old('state', $address->state ?? '') }}',
        initCityName:     '{{ old('city', $address->city ?? '') }}',
        initDistrictName: '{{ old('district', $address->district ?? '') }}'
     })" x-init="init()">
    <select x-model="stateId" @change="onStateChange()">
        <option value="">Pilih provinsi</option>
        <template x-for="p in provinces" :key="p.id">
            <option :value="p.id" x-text="p.name"></option>
        </template>
    </select>
    <input type="hidden" name="state" :value="stateName">
    ...
</div>
```

Alasan pakai `<input type="hidden" name="state">`: kolom DB tetap menyimpan nama wilayah (string), bukan id. Jadi UX user tinggal pilih dari dropdown, tapi yang dikirim ke backend tetap nama.

#### b. Perketat RT / RW

Ganti `<input type="text">` jadi:

```html
<input type="text" name="rt" inputmode="numeric" maxlength="3"
       pattern="\d{1,3}" value="{{ old('rt') }}"
       oninput="this.value = this.value.replace(/\D/g, '').slice(0, 3)">
```

- `inputmode="numeric"` → mobile keyboard langsung numerik.
- `maxlength="3"` dan `pattern="\d{1,3}"` → batasi client-side.
- `oninput` → filter semua karakter non-digit secara realtime.

#### c. Perketat Kode Pos

```html
<input type="text" name="zip_code" inputmode="numeric" maxlength="5"
       pattern="\d{5}" value="{{ old('zip_code') }}"
       oninput="this.value = this.value.replace(/\D/g, '').slice(0, 5)">
```

### 3. JS Helper: `resources/js/address-wilayah.js` (file baru)

Define Alpine component global:

```js
document.addEventListener('alpine:init', () => {
    Alpine.data('addressWilayah', (opts) => ({
        BASE: 'https://emsifa.github.io/api-wilayah-indonesia/api',
        provinces: [],
        cities: [],
        districts: [],
        stateId: opts.initStateId || '',
        cityId:  opts.initCityId  || '',
        districtId: opts.initDistrictId || '',
        stateName: opts.initStateName || '',
        cityName:  opts.initCityName  || '',
        districtName: opts.initDistrictName || '',

        async init() {
            this.provinces = await fetch(`${this.BASE}/provinces.json`).then(r => r.json());
            if (this.stateId) {
                this.cities = await fetch(`${this.BASE}/regencies/${this.stateId}.json`).then(r => r.json());
            }
            if (this.cityId) {
                this.districts = await fetch(`${this.BASE}/districts/${this.cityId}.json`).then(r => r.json());
            }
        },

        async onStateChange() {
            this.cityId = ''; this.districtId = '';
            this.cities = []; this.districts = [];
            const p = this.provinces.find(x => x.id === this.stateId);
            this.stateName = p?.name ?? '';
            if (this.stateId) {
                this.cities = await fetch(`${this.BASE}/regencies/${this.stateId}.json`).then(r => r.json());
            }
        },

        async onCityChange() {
            this.districtId = ''; this.districts = [];
            const c = this.cities.find(x => x.id === this.cityId);
            this.cityName = c?.name ?? '';
            if (this.cityId) {
                this.districts = await fetch(`${this.BASE}/districts/${this.cityId}.json`).then(r => r.json());
            }
        },

        onDistrictChange() {
            const d = this.districts.find(x => x.id === this.districtId);
            this.districtName = d?.name ?? '';
        },
    }));
});
```

Import di `resources/js/app.js`:

```js
import './address-wilayah';
```

### 4. Fallback UX

- Kalau fetch API gagal (misal offline), tampilkan error kecil di bawah dropdown dan tetap allow submit dengan nilai lama (saat edit) tapi block submit kalau provinsi belum terpilih (HTML `required` di select).
- Saat edit, nilai lama (string nama wilayah) tetap ditampilkan di option placeholder sehingga user tidak kebingungan.

2. Coba submit dengan RT `abc` → ditolak client-side (tidak bisa diketik).

## Risiko & Catatan

- API emsifa adalah static JSON di GitHub Pages, uptime bagus tapi bukan SLA. Kalau khawatir ketergantungan, bisa di-fase 2 migrate ke seed lokal.
- Schema DB tetap `string` untuk state/city/district, tidak ada migrasi. Jadi aman dari sisi data lama.
- Kalau ke depan Anda butuh data lebih granular (kelurahan/desa), tinggal tambahkan level ke-4 ke component yang sama dengan endpoint `/villages/{district_id}.json`.
