## Executive Summary

**Lanina Patisserie** adalah sistem e-commerce berbasis Laravel 11 untuk toko kue/patisserie dengan fitur pre-order, manajemen promosi, dan sistem rekomendasi berbasis algoritma Apriori. **⚠️ PENTING - Algoritma Rekomendasi Saat Ini**:

**BUKAN Apriori Algorithm**, melainkan **Manual Association Rules Mining**:

**Implementasi di `PromoController::buildPromoCombinationRanking()` (lines 67-169)**:

1. **Data Source**: Transaksi dari `PreOrderDetail` dengan status valid (processing/shipping/completed)

2. **Algoritma**:
   ```
   - Ekstrak transaksi yang memiliki >= 2 produk
   - Hitung frekuensi individual produk (productFrequency)
   - Hitung frekuensi pasangan produk (pairFrequency) dengan nested loop

   ```

3. **Metrik yang Dihasilkan**:
   - **Support**: Proporsi transaksi yang mengandung pasangan produk
   - **Confidence A→B**: Probabilitas B dibeli ketika A dibeli
   - **Confidence B→A**: Probabilitas A dibeli ketika B dibeli
   - **Lift**: Rasio support vs expected random co-occurrence
   - **Score**: Weighted combination untuk ranking

4. **Limitasi**:
   - ❌ Hanya menangani **2-itemsets** (pasangan produk)
   - ❌ Tidak ada iterative candidate generation (ciri khas Apriori)
   - ❌ Tidak ada minimum support threshold filtering
   - ❌ Tidak bisa menemukan 3+ item bundles
   - ❌ Kompleksitas O(n²) untuk setiap transaksi

5. **Endpoint**: `/admin/promo/rekomendasi`

## 14. TEMUAN PENTING: Algoritma Rekomendasi

### ❌ Bukan Apriori Algorithm

Setelah memeriksa `app/Http/Controllers/PromoController.php`, saya menemukan bahwa:

**Implementasi saat ini BUKAN Apriori algorithm**, melainkan **simplified association rules mining** dengan karakteristik:

### Implementasi Aktual

**Method**: `PromoController::buildPromoCombinationRanking()` (lines 67-169)

**Algoritma**:
```php
1. Ambil transaksi dari PreOrderDetail (status: processing/shipping/completed)
2. Filter transaksi yang memiliki >= 2 produk
3. Hitung frekuensi individual produk
4. Nested loop untuk hitung frekuensi pasangan produk:
   for i = 0 to n-1:
     for j = i+1 to n:
       count pair (product[i], product[j])
5. Hitung metrik untuk setiap pasangan:
   - Support = pairCount / totalTransactions
   - Confidence A→B = pairCount / frequencyA
   - Confidence B→A = pairCount / frequencyB

6. Weighted score = (support × 0.4) + (max_confidence × 0.3) + (lift/3 × 0.3)
7. Return top 10 kombinasi
```

### Perbedaan dengan Apriori

| Aspek | Implementasi Saat Ini | Apriori Algorithm |
|-------|----------------------|-------------------|
| **Itemset Size** | Hanya 2-itemsets (pasangan) | 1, 2, 3, ..., n-itemsets |
| **Candidate Generation** | Brute force nested loop | Iterative join & prune |
| **Min Support Threshold** | Tidak ada filtering | Filter di setiap iterasi |
| **Scalability** | O(n²) per transaksi | O(2^n) tapi dengan pruning |
| **Output** | Top 10 pasangan produk | All frequent itemsets |
| **Association Rules** | Dihitung langsung | Generated dari frequent itemsets |

### Limitasi Implementasi Saat Ini

1. **Tidak bisa menemukan 3+ item bundles**
   - Contoh: "Kue A + Kue B + Kue C" tidak terdeteksi
   - Hanya bisa: "Kue A + Kue B"

2. **Tidak ada minimum support filtering**
   - Semua pasangan dihitung, bahkan yang jarang muncul
   - Bisa menghasilkan rekomendasi yang tidak signifikan

3. **Performance issue untuk dataset besar**
   - Nested loop O(n²) untuk setiap transaksi
   - Tidak ada early pruning

4. **Tidak ada iterative candidate generation**
   - Ciri khas Apriori: generate k+1 itemsets dari k-itemsets
   - Implementasi saat ini: langsung hitung semua pasangan

## 15. PLAN: Update Dokumentasi - Association Rules Mining

### Tujuan
Membuat dokumentasi baru yang akurat untuk menjelaskan sistem rekomendasi promosi yang sebenarnya digunakan (Association Rules Mining untuk 2-itemsets), menggantikan referensi ke Apriori algorithm yang tidak diimplementasikan.

### Scope Pekerjaan

#### 1. Buat File Dokumentasi Baru
**File**: `RECOMMENDATION_SYSTEM.md` (root directory)

**Struktur Konten**:

```markdown
# Sistem Rekomendasi Promosi - Association Rules Mining

## Overview
Sistem ini menggunakan **Association Rules Mining** untuk menemukan pola pembelian produk yang sering dibeli bersama, kemudian merekomendasikan kombinasi produk untuk bundle promosi.

## Metode: 2-Itemset Association Rules

### Konsep Dasar
- Analisis transaksi historis dari PreOrder
- Menemukan pasangan produk yang sering dibeli bersama
- Menghitung metrik statistik untuk ranking kombinasi

### Metrik yang Digunakan

1. **Support**: Proporsi transaksi yang mengandung pasangan produk
   - Formula: `support(A,B) = count(A ∩ B) / total_transactions`
   - Contoh: 15 dari 100 transaksi = 15% support

2. **Confidence**: Probabilitas kondisional
   - Confidence A→B: `P(B|A) = count(A ∩ B) / count(A)`
   - Confidence B→A: `P(A|B) = count(A ∩ B) / count(B)`
   - Contoh: 15 dari 20 transaksi yang beli A juga beli B = 75% confidence

3. **Lift**: Rasio co-occurrence vs random
   - Formula: `lift = support(A,B) / (support(A) × support(B))`
   - Lift > 1: produk saling mendukung
   - Lift = 1: independen
   - Lift < 1: saling menghambat

4. **Weighted Score**: Kombinasi metrik untuk ranking
   - Formula: `(support × 0.4) + (max_confidence × 0.3) + (lift/3 × 0.3)`
   - Prioritas: support tinggi + confidence tinggi + lift positif

## Implementasi Teknis

### Lokasi Code
- **Controller**: `app/Http/Controllers/PromoController.php`
- **Method**: `buildPromoCombinationRanking()` (lines 67-169)
- **View**: `resources/views/pages/promo-admin/rekomendasi.blade.php`
- **Route**: `/admin/promo/rekomendasi`

### Algoritma

```
1. Ekstrak Data Transaksi
   - Ambil PreOrderDetail dengan status: processing, shipping, completed
   - Group by pre_order_id
   - Filter transaksi dengan >= 2 produk

2. Hitung Frekuensi Individual
   - Loop setiap transaksi
   - Count berapa kali setiap product_id muncul
   - Simpan di array productFrequency

3. Hitung Frekuensi Pasangan
   - Nested loop untuk setiap transaksi:
     for i = 0 to n-1:
       for j = i+1 to n:
         pairKey = min(product[i], product[j]) + ":" + max(product[i], product[j])
         pairFrequency[pairKey]++

4. Hitung Metrik untuk Setiap Pasangan
   - Support = pairCount / totalTransactions
   - Confidence A→B = pairCount / frequencyA
   - Confidence B→A = pairCount / frequencyB
   - Lift = support / (P(A) × P(B))
   - Score = weighted combination

5. Ranking & Output
   - Sort by score descending
   - Return top 10 kombinasi
```

### Kompleksitas
- **Time**: O(T × N²) dimana T = jumlah transaksi, N = rata-rata produk per transaksi
- **Space**: O(P + C) dimana P = jumlah produk unik, C = jumlah pasangan unik

## Cara Menggunakan

### 1. Akses Halaman Rekomendasi
```

```
## Limitasi & Batasan

### Limitasi Teknis
1. **Hanya 2-Itemsets**
   - Tidak bisa detect bundle 3+ produk
   - Contoh: "Kue A + Kue B + Kue C" tidak terdeteksi

2. **Tidak Ada Threshold Filtering**
   - Semua pasangan dihitung, termasuk yang jarang
   - Bisa menghasilkan rekomendasi tidak signifikan

3. **Performance untuk Dataset Besar**
   - Nested loop O(N²) per transaksi
   - Untuk ribuan transaksi bisa lambat

4. **Real-time Calculation**
   - Dihitung on-the-fly setiap page load
   - Tidak ada caching atau pre-computation

### Batasan Bisnis
1. **Hanya Transaksi Valid**
   - Status: processing, shipping, completed
   - Unpaid/cancelled tidak dihitung

2. **Minimum 2 Produk per Transaksi**
   - Single-item orders diabaikan
   - Bisa bias untuk produk populer

3. **Top 10 Only**
   - Hanya menampilkan 10 kombinasi terbaik
   - Kombinasi lain tidak terlihat

## Interpretasi Hasil

### Contoh Output
```
#1 Chocolate Cake + Vanilla Cupcake
Support: 12.5% | Confidence: 60% | Lift: 2.4x

Artinya:
- 12.5% dari semua transaksi membeli kedua produk ini
- 60% pelanggan yang beli Chocolate Cake juga beli Vanilla Cupcake
- Kombinasi ini 2.4x lebih sering terjadi dibanding random
```

### Rekomendasi Bisnis
- **High Support + High Confidence**: Bundle populer, pasti laku
- **Low Support + High Confidence**: Niche bundle, target spesifik
- **High Lift**: Produk saling melengkapi, good bundle
- **Low Lift**: Produk independen, kurang cocok bundle

## Troubleshooting

### Q: Tidak ada rekomendasi yang muncul?
**A**: Kemungkinan penyebab:
- Belum ada transaksi dengan status valid
- Semua transaksi hanya 1 produk
- Database kosong

**Solusi**: Pastikan ada minimal 5-10 transaksi dengan 2+ produk

### Q: Rekomendasi tidak masuk akal?
**A**: Kemungkinan penyebab:
- Data transaksi terlalu sedikit
- Produk terlalu banyak variasi
- Tidak ada pola pembelian yang jelas

**Solusi**: 
- Tunggu lebih banyak data transaksi
- Filter produk berdasarkan kategori
- Adjust weighted score formula

### Q: Performance lambat?
**A**: Kemungkinan penyebab:
- Terlalu banyak transaksi historis
- Nested loop O(N²) per transaksi

**Solusi**:
- Limit transaksi ke 3-6 bulan terakhir
- Implement caching dengan Laravel Cache
- Pre-compute di background job
```

```php
/**
 * Build ranked product-pair recommendations using Association Rules Mining.
 *
 * This method analyzes historical transaction data to find frequently 
 * co-purchased product pairs and ranks them using statistical metrics.
 *
 * Algorithm: 2-Itemset Association Rules (NOT Apriori)
 * - Only handles pairs of products (2-itemsets)
 * - No iterative candidate generation
 * - No minimum support threshold filtering
 *
 * Metrics calculated:
 * - support(A,B): P(A and B) = proportion of transactions containing both
 * - confidence(A->B): P(B|A) = probability of B given A was purchased
 * - confidence(B->A): P(A|B) = probability of A given B was purchased
 * - lift: support(A,B) / (support(A) * support(B)) = co-occurrence ratio
 * - score: weighted combination for ranking
 *
 * Complexity: O(T * N^2) where T = transactions, N = avg items per transaction
 *
 * @return Collection<int, array{
 *   product_ids: array<int>,
 *   products: array<string>,
 *   support: float,
 *   confidence_a_to_b: float,
 *   confidence_b_to_a: float,
 *   lift: float,
 *   score: float
 * }> Top 10 product pair recommendations sorted by score
 */
```
